\section{实验与结果分析}

大语言模型评测需要结合任务场景、评价指标和输出可靠性设计评价口径\cite{luowen2024_llm_eval}。本章围绕基于大模型的课程知识实体识别与关系抽取系统，报告实验设计与运行结果。先介绍实验环境与配置参数，随后从课程语料构建、实体抽取评估、关系抽取评估、零样本与少样本抽取对比评估、知识图谱质量分析以及典型案例分析等角度，对系统运行结果进行评估与讨论。

\subsection{零样本与少样本抽取对比评估}

\subsubsection{评估集与匹配口径}
\label{sec:5-zsfs-protocol}

为了让zero-shot（zs）与few-shot（fs）的抽取结果可以进行公平比较，本实验以\texttt{data/chunks}中的原始文本块为回溯对象，构建并复核独立评估集。用于评估的金标文件为\texttt{data/eval/gold\_triples.json}，包含500条关系三元组和594个实体，覆盖88个源文件以及186个源文本块。核查过程中逐条检查了\texttt{source\_file}、\texttt{chunk\_index}与\texttt{evidence}字段：所有源文件均存在，所有chunk索引均有效，500条证据文本均为对应源文本块中的连续原文子串，其中17条证据已替换为保留原始换行的精确片段。

KG抽取评估面临的一个共性困难是：相同语义的实体在原文中可能出现多种表述（中英、缩写、含括号注释等），而严格五字段相等会把这些"语义对、字面错"的命中全部计为假阴性。为兼顾可比性与可信度，本实验对500条金标采用四档复合匹配口径：

\begin{itemize}
  \item \textbf{严格匹配（Strict）}：原始\texttt{kgeval}口径，\texttt{(head, head\_type, relation, tail, tail\_type)} 五字段完全相等。作为保守下界。
  \item \textbf{软匹配（Substring）}：实体名先做NFKC归一化、去除括号注释、大小写折叠、空白与标点折叠；类型与关系仍要求精确相等；归一化后若预测实体名是金标实体名的子串、或反之，则认为该端点匹配。该口径用于在不放松类型约束的前提下，吸收"TCP" / "传输控制协议(TCP)" / "TCP拥塞控制"这类同指但字面不同的表述。
  \item \textbf{槽位匹配（Head$+$Relation）}：在软匹配基础上仅约束\texttt{head}名、\texttt{head\_type}、\texttt{relation}、\texttt{tail\_type}四项，允许\texttt{tail}名不同。该口径回答"模型是否对正确头实体与关系类型给出了一个合理的尾实体"，与槽位填充式评估口径一致。
  \item \textbf{类型级（Relation-level）}：仅匹配\texttt{(head\_type, relation, tail\_type)} 三元，反映知识图谱模式（schema）层面的覆盖能力，作为上界。
\end{itemize}

为避免用全量图谱去对比500条抽样金标时把大量未标注但可能正确的关系误计为假阳性，实验先把四种raw抽取输出过滤到金标覆盖的186个源文本块，再在关闭embedding合并的条件下执行\texttt{kgmerge}，最后用同一份金标计算指标。该口径把比较范围限制在相同文本证据范围内，适合观察zs/fs提示方式和复核开关对各档命中率的影响。

表~\ref{tab:5-zsfs}中"实体F1$_\text{N}$"为名称归一化下的实体F1，"R$_\text{rel}$"为类型级召回率。从严格档到类型级档，三元组F1按"严格 $\to$ 软匹配 $\to$ 槽位 $\to$ 类型级"的顺序系统性升高（以\texttt{zs-recheck}为例：13.76\% $\to$ 18.40\% $\to$ 42.49\% $\to$ 81.93\%），这一阶梯说明系统的失分主要并非源于关系类型判断错误，而是源于端点实体名称的表述差异：在保持类型与关系正确的前提下，仅将实体名匹配条件由"严格相等"放宽为"子串包含"就可使三元组F1相对提升约35\%~50\%；进一步放宽到只看头实体则可达35\%~42\%。这一结果与课程材料中频繁出现的中英缩写并用现象（如TCP / 传输控制协议、IP地址 / IP address、CSMA/CA / 二进制指数退避）相吻合。

各匹配档下的相对优劣可从两个角度观察。其一，在严格档下\texttt{zs-nocheck}的三元组F1最高（15.29\%），但优势随匹配口径放宽逐渐缩小，到槽位档时\texttt{zs-recheck}反超达到42.49\%，到类型级档时\texttt{zs-recheck}进一步以81.93\%领先全部配置。其二，复核（recheck）整体上把模式从"高字面重合"推向"高语义覆盖"——\texttt{zs-recheck}相对\texttt{zs-nocheck}虽然损失了少量严格命中，但在槽位与类型级两档显著提升；\texttt{fs-recheck}相对\texttt{fs-nocheck}主要把实体F1$_\text{N}$从32.89\%抬至34.16\%，而严格档基本持平。这与复核机制的设计目标一致：通过LLM对每条候选关系做证据回看，倾向于剔除字面对但语义弱的命中，并保留语义对而字面表述更自然的候选。

需要特别指出的是，本实验中的金标是经人工复核的正例参考集，并非对186个源文本块中所有可能正确关系的穷尽标注。因此，表中的精确率应理解为相对于这一抽样金标的保守估计；如果以系统抽取的全部"语义正确"三元组为分子，实际生产精确率会高于表中数字。

raw层指标整体高于合并层，原因是合并阶段会把同一实体的不同表面形式归并为一个规范名称，反而可能把原本对得上金标的字面命中替换为更"标准"但未必与金标完全相同的形式，造成轻微的字面错配。这一现象在严格档与软匹配档体现尤为明显（如\texttt{zs-nocheck}严格三元组F1从合并层15.29\%升至raw层17.40\%）；而到了槽位与类型级档，差异基本消失，说明合并对"关系类型与头实体的整体识别能力"几乎无损。

属性分类方面，ZS的head\_type准确率（86.67\%）与FS（84.93\%）接近，但FS在tail\_type上显著领先（89.04\% vs 75.24\%，$+$13.80个百分点），relation准确率两者相当（85.71\% vs 86.30\%）。覆盖方面，ZS在186个金标块中找到182个对应raw文件（缺失4个），FS找到176个（缺失10个）。

综合两层结果，可以得到三个结论：

\begin{enumerate}
    \item \textbf{严格F1并不能单独刻画系统能力}。严格档下三元组F1普遍在11\%~17\%区间，但软匹配档可至15\%~20\%，槽位档可至34\%~42\%，类型级档可至74\%~82\%。从严格到类型级的阶梯显示，模型在关系类型与头实体识别上具备较强能力（可达80\%量级），主要丢分集中在实体表面形式与原文/金标的字面差异。这一观察支持把软匹配与槽位匹配作为KG抽取评估的主指标，并把严格匹配作为保守下界。
    \item \textbf{复核开关在不同匹配档下作用方向不同}。在严格档下，复核往往略微降低三元组F1（剔除字面命中）；但在槽位与类型级档，复核普遍提升F1，尤其是\texttt{zs-recheck}在合并层类型级F1上达到81.93\%，全场最高。这表明复核的价值不在于"提高字面准确率"，而在于"用语义判断替换字面命中"，因此在对实体表述宽容的口径下才能体现优势。
    \item \textbf{Few-shot需配合金标风格而非简单加示例}。在当前提示词下，FS仅在tail\_type分类（$+$13.80个百分点）和合并层严格精确率个别配置上略胜ZS，但在多数综合指标上不及ZS，且抽取缺失chunk明显更多。后续优化方向不应是单纯增加few-shot示例数量，而应聚焦少量但高质量的金标风格对齐示例。
\end{enumerate}

\subsection{本章小结}

本章对系统进行了实验评估。系统构建出包含32,760个实体节点和17,627条关系边的课程知识图谱（\texttt{merged-fs-recheck}），实体分布以概念类型为主（69.3\%），关系分布以contains和applied\_to为主（69.8\%），连通子图平均度数2.71。本章把严格五字段匹配口径（原\texttt{kgeval}口径）扩展为"严格／软匹配／槽位／类型级"四档评估，结果显示：在合并层面，类型级三元组F1可达74\%~82\%，槽位F1可达37\%~42\%，软匹配F1可达15\%~19\%，严格F1则在11\%~16\%。从严格到类型级的阶梯说明系统的失分主要源于实体名称的表面形式差异，而非关系类型判断错误，这与课程材料中频繁出现的中英缩写并用现象一致。零样本与少样本对比表明，zero-shot在多数综合指标下整体优于few-shot，复核（recheck）开关在严格档下略微降低F1、但在槽位与类型级档下普遍提升F1（\texttt{zs-recheck}在合并层类型级F1达到81.93\%，全场最高），few-shot仅在tail\_type分类上具有局部优势（89.04\% vs 75.24\%）。后续优化方向应聚焦少量高质量的金标风格对齐示例，而非单纯增加few-shot示例数量。
