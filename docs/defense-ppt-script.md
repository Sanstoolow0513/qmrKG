# QmrKG 论文答辩 PPT 文本（~10分钟）

---

## 第1页：封面

**标题：** QmrKG —— PDF 到知识图谱的端到端自动化 Pipeline

**副标题：** 面向计算机网络课程教材的知识提取、融合与可视化系统

**口述（约30秒）：**

各位老师好，我的毕业设计题目是 QmrKG，一个把 PDF 教材自动转成知识图谱的端到端系统。简单说就是：把一本几百页的计算机网络教材扔进去，系统自动识别文字、抽取知识点和它们之间的关系，最后生成一张可以交互浏览的知识网络图。整套流程从 PDF 解析到前端可视化，全部自动化。

---

## 第2页：问题背景 & 动机

**要点：**
- 计算机网络教材知识点密集、概念关系复杂
- 学生复习时：翻书找关联 = 线性阅读，效率低
- 如果把知识组织成图结构，一眼就能看到"这个协议依赖哪些机制、和哪些协议对比"

**口述（约40秒）：**

为什么要做这个？计算机网络这门课有个特点——知识点特别多，而且关系复杂。比如 TCP 这个协议，它涉及拥塞控制、三次握手、流量控制，又和 UDP 是对比关系。学生复习的时候，翻书只能线性阅读，很难建立全局的关联认知。如果把知识组织成图结构，你点一个节点就能看到所有关联——这比翻书高效得多。但问题是：手工构建知识图谱太费人了，所以我的目标是自动化。

---

## 第3页：系统总览

**流程图（7 阶段）：**
```
PDF → PNG → Markdown(OCR) → 整书MD合并 → JSON分块 → 三元组抽取 → 合并去重 → Neo4j → 前端可视化
```

**处理规模：** 205 个 PDF + 69 个 PPT → 22,666 张图片 → 57M 字符 Markdown → 15,367 个文本块 → 60 万实体 + 20 万三元组（原始）

**口述（约1分钟）：**

整个系统分七个阶段。第一步，把 PDF 或 PPT 转成图片，用 PyMuPDF 渲染，PPT 的话先通过 LibreOffice 转 PDF。第二步，用一个多模态视觉大模型 qwen3-vl-8b 做 OCR，把图片转成带格式的 Markdown，保留标题层级、编号结构。第三步，因为 OCR 是按页输出的，需要把同一本书的所有页面合并成一个完整的 Markdown 文件。第四步，把长文本按 token 切块，每块不超过 4000 token，确保扔给大模型时不会超上下文窗口。第五步是核心：用 deepseek-v4-flash 从每个文本块里抽取实体和三元组——比如识别出"TCP"是 protocol、"拥塞控制"是 concept，它们之间的关系是"applied_to"，并且必须从原文中给出 evidence 证据句。第六步，把海量 raw 三元组合并去重——同名实体归一化、冲突关系消除。最后导入 Neo4j 图数据库，前端用 Next.js + react-force-graph 做力导向图可视化。实际处理规模：205 个 PDF、69 个 PPT，产生了 60 万个实体和 20 万条三元组。

---

## 第4页：核心技术决策 & 架构

**要点：**
- LLM 工厂模式：统一管理 4 个大模型（视觉 OCR、文本抽取、文本审核、Embedding）
- 速率限制：RollingRateLimiter，支持 rpm + 并发数双重控制
- 完整配置驱动：config.yaml 控制一切，Prompt 可通过 YAML 切换（zs/fs/review）
- 双模式抽取：Zero-shot vs Few-shot，可切换对比

**口述（约50秒）：**

架构上，我做了一个统一的 LLM 工厂层。系统调用了四个不同模型——视觉模型做 OCR、文本模型做抽取、Embedding 模型做实体归一化、还有一个审核模型做质量检查。所有这些调用走统一的工厂接口，集成了速率限制和重试机制，不会因为 API 限流而崩溃。另一个设计选择是：所有 prompt 和模型配置都在 config.yaml 里集中管理，切换 zero-shot 和 few-shot 抽取模式只需要改一行配置。这为后续的实验对比提供了很大便利。

## 第5页：最难的问题 —— 实体归一化

**要点（为什么要专门讲这个）：**
- 同一个实体在不同 chunk 里叫法不同："传输控制协议" vs "TCP" vs "TCP 协议"
- 传统方案：规则匹配（别名表）→ 只能覆盖已知的
- 我的方案：三层归一化流水线
  1. 规则别名表（27 条，覆盖常见中英文对照）
  2. Embedding 语义匹配（FAISS 向量检索 + 余弦相似度，阈值 0.85）
  3. LLM 二次复核（只有 Embedding 候选对的才送给 LLM 做最终裁决）
- 用 Union-Find 做聚类传递闭包

**口述（约1分10秒）：**

整个系统里最难的问题不是调用大模型，而是实体归一化。同一个概念在不同地方有不同叫法——比如 TCP，有的地方叫"传输控制协议"，有的地方直接叫"TCP"，有的叫"TCP 协议"。如果不归一化，知识图谱里就会出现三个重复节点。

我设计了一个三层流水线来解决这个问题。第一层是规则别名表，覆盖了 27 条常见中英文对照，比如"超文本传输协议"→"HTTP"，这部分是零成本的精确匹配。第二层用 Embedding 模型把所有实体名转成向量，通过 FAISS 做近似最近邻检索，余弦相似度超过 0.85 的作为候选对。第三层最关键——把候选对送给 LLM 做二次复核，LLM 会根据两个实体在原文里的上下文（三元组、描述）来判断它们到底是不是同一个东西。最后用 Union-Find 做聚类传递闭包，保证如果 A=B 且 B=C，那 A=C 也成立。

这个三层设计的好处是：规则层零成本覆盖最常见的，Embedding 层低成本过滤海量的，LLM 层高精度裁决边缘的——既保证了质量，又控制了成本。

## 第6页：如何评估 —— 评测体系

**要点：**
- Gold standard：500 条人工审核三元组，覆盖 56 个 chunk、47 个源文件、8 类资料
- 来源分布：中文教材 22、英文教材 9、中文参考书 6、RFC 协议文档 7、作业答案 3、实验 3、复习资料 3、其他 3
- 双重评估体系：
  - **Raw eval**（抽取前）：ZS vs FS 对比，6 维度
  - **Merged eval**（合并后）：严格匹配 P/R/F1 + evidence 覆盖率
- 匹配粒度：纯严格匹配（exact match），不做 fuzzy

**口述（约1分钟）：**

评估这块我花了很大功夫。首先我建了一个 gold standard——从 14240 个文本块里抽样了 56 个，覆盖 8 类资料：中文教材、英文教材、中文参考书、RFC 协议文档、作业答案、实验指导、复习资料，确保代表性。然后我写了一个 agent 做独立标注，每条 triple 都必须附带 evidence（原文证据句），然后我逐条人工审核，修正了 17 条 evidence 使其和原文完全一致。最终是 500 条 triple，594 个实体。

评估体系分两层。第一层是 raw 评估，直接测每 chunk 的抽取质量，六个维度：实体 F1、三元组 F1、关系 F1（忽略实体名只看关系类型），再加 head_type、tail_type、relation 三个属性的准确率。这样做的好处是：关系 F1 可以排除"实体名叫法不同但关系对"的干扰。第二层是 merged 评估，测合并去重后的最终质量。

我选择纯严格匹配而不是 fuzzy matching，是为了保证可复现——任何人在任何时间跑同样的评估，结果完全一致。

## 第7页：实验结果

**要点：**

| 维度 | ZS | FS | 结论 |
|------|-----|-----|------|
| 实体 F1 | 33.4% | 31.4% | ZS 略优 |
| 三元组 F1 | 13.9% | 11.3% | ZS 略优 |
| **关系 F1** | **65.4%** | 60.2% | ZS 显著优于 FS |
| head_type 准确率 | 86.7% | 84.9% | 均较高 |
| tail_type 准确率 | 75.2% | 89.0% | FS 反超 |
| relation 准确率 | 85.7% | 86.3% | 接近 |

**口述（约1分钟）：**

这是 raw 评估的 ZS vs FS 对比结果。先说最重要的发现：关系 F1——不看实体名只看关系——ZS 达到 65.4%，FS 是 60.2%。这说明大模型对"TCP 和 UDP 是 compared_with 关系"这类判断非常准确。但为什么三元组 F1 只有 14%？因为严格匹配要求实体名也完全一致。模型抽取的是"传输控制协议"，gold 里写的是"TCP"，就算关系完全对，也算不匹配。这个差距不代表抽取质量差——属性准确率都在 75-89%，证明分类能力是可靠的。

第二个发现：Zero-shot 总体优于 Few-shot。FS 的示例可能引入了过拟合 bias，让模型变得保守。这也反过来说明 prompt 设计比堆示例更关键。

第三个发现：FS 在 tail_type 上反超 ZS，说明示例对分类任务有帮助，但对关系判断反而是负面影响。

## 第9页：预判 Q&A —— 答辩可能会被问的问题

### Q1: "三元组 F1 才 14%，这个系统能用吗？"

**回答思路：**
- 14% 是严格匹配的结果，不代表抽取质量差
- 关系 F1 是 65%，说明模型理解内容是准确的
- 问题的根源是实体名表述不一致（"传输控制协议" vs "TCP"），这不是抽取能力问题，是归一化问题
- 这正是为什么我花了最大精力在 kg_merger 的实体归一化上
- 另外，这个数据是基于 raw extraction（合并前）的——实际经过 kgmerge 归一化后，很多 FP/FN 会被消除

### Q2: "Gold standard 是你自己标的，怎么保证客观？"

**回答思路：**
- 标注过程有严格规范：每条 triple 必须附带原文 evidence，证据句必须是连续原文子串
- 我做了 evidence 验证：逐条检查 500 条 evidence 是否在源 chunk 中存在精确匹配，修正了 17 条
- 有 rejection log 记录被拒绝的 triple 及原因（evidence 不包含实体名、证据太弱等）
- 诚实地说：单标注者确实是一个局限，未来可以引入多人标注 + inter-annotator agreement
- 但目前的 gold 是 source-backed 的——每一条都能追溯到原文，这是客观的底线

### Q3: "你这个和直接用 ChatGPT 问有什么区别？"

**回答思路：**
- ChatGPT 只能一问一答，不能处理几百页的 PDF 教材
- 我的系统是 pipeline：自动切块 + 批量抽取 + 去重归一化 + 入库 + 可视化
- 质量控制不一样：我有专门的 review prompt 审核每条 triple，检查 evidence 是否真的在原文里、head/tail 是否都出现在 evidence 中
- 可复现性：同样的 PDF + 同样的配置 → 同样的结果（deterministic 的 pipeline 部分）
- 最重要的是评测闭环：我有 gold standard 和评估体系，知道模型哪里好哪里差，ChatGPT 你没法量化评估它的输出质量

### Q4: "创新点在哪？"

**回答思路：**
- 创新不在"调用大模型"这个动作本身，而在系统设计和评测方法
- 三层实体归一化流水线（规则+Embedding+LLM复核）：结合了确定性规则和语义匹配，比纯规则或纯 Embedding 都更可靠
- 双重评估体系（raw+merged，6 维度）：不是简单的"准不准"，而是拆解到关系层、属性层，能定位到具体哪个环节出问题
- Evidence 闭环：从抽取（必须带 evidence）到审核（验证 evidence 在原文中）到评估（计算 evidence 覆盖率），全链路可追溯
- 完整工程：从 PDF 到可视化，单一命令 `uv run qmr` 跑全流程

### Q5: "为什么选这些模型？有没有做过对比？"

**回答思路：**
- OCR 选 qwen3-vl-8b：支持中文、多模态、能识别标题层级，速度够快
- 抽取选 deepseek-v4-flash：支持 thinking 模式、中文能力强、性价比高
- Embedding 选 qwen3-embedding-8b：1024 维向量，中文语义匹配好
- ZS vs FS 本身就是一次模型行为对比——结果很反直觉：ZS 反而更好
- 未来可以加入更多模型的横向对比（如 GPT-4o、Claude），但目前的选型是基于可用性、成本和中文能力的平衡

### Q6: "这个系统有什么实际用途？能不能落地？"

**回答思路：**
- 定位是教育辅助工具，不是生产级产品
- 实际场景：老师上传课件 PDF → 自动生成知识图谱 → 学生可视化浏览复习
- 局限：仅限计算机网络领域（prompt 是针对这个领域写的）、对公式/图表支持有限、依赖 LLM API（有成本和延迟）
- 可扩展性：换一套 prompt 就能适配其他学科；OCR 和抽取是解耦的，可以单独升级

### Q7: "和已有的知识图谱构建工具（DeepKE、OpenIE）比有什么优势？"

**回答思路：**
- DeepKE/OpenIE 是通用 NLP 工具，面向新闻/百科等规范文本，对教材这种混合中英文、有公式图表、排版复杂的文档效果很差
- 它们不做 OCR——输入必须是纯文本，而我的系统从 PDF 像素开始处理
- 它们不做实体归一化——抽取完就结束了，不会合并"TCP"和"传输控制协议"
- 它们没有 evidence 机制——你无法追溯每条三元组是从原文哪句话来的
- 我的系统不是要替代它们，而是解决它们覆盖不到的场景：从非结构化教学文档到结构化知识图谱的端到端自动化
- 另外，我的评估体系可以告诉你系统到底有多准——这是很多 KG 工具缺失的
